05 · How a run works — in plain language

Six steps from idea to verdict.

  1. Understand the askAiki asks you 3–4 sharp clarifying questions, then locks a "decision contract": the exact decision, your constraints, and what success means. No vague prompts go downstream.
  2. Independent researchTwo scout models work the problem alone in separate lanes — one on market & adoption, one on economics & delivery. Neither sees the other's answer, so no groupthink.
  3. Build the claim graphEvery claim, every piece of evidence, every stance becomes a typed graph — so agreement and genuine disagreement are detected structurally, not by skimming prose.
  4. Cross-examineLoad-bearing claims get verified; real disagreements get a bounded rebuttal round — concede, counter, or narrow. Weak claims die here, not in your product.
  5. The chairman rulesThe strongest model sits as judge: each disputed claim HOLDS, FAILS, or stays UNRESOLVED — rolled into one recommendation with explicit conditions.
  6. Report + receiptsYou get a decision report: verdict, confidence score, validation plan, open questions — plus a receipt of every call, token and second spent.

08 · How to use it

Install once, convene anytime.

$ npm install -g aiki-cli
$ aiki serve # browser workspace
$ aiki       # or the terminal TUI
  • Type your ideaPlain language: "Should I build X or Y?" Pick quick, council, or let auto route it.
  • Answer the grill3–4 clarifying questions sharpen your idea before any model spends a token on it.
  • Approve the spendAiki shows the estimated calls up front; nothing paid runs without your yes. File and URL access are gated the same way.
  • Read the verdict, ask follow-upsThe report lands in the workspace; follow-up questions cost exactly one call — labeled "no council".
